단일 HTML에서 호텔 ERP 플랫폼 구조로

현재 연회 ERP는 빠른 프리뷰 개발에는 적합하지만, 기능이 한 파일에 집중되어 유지보수가 어려워지고 있습니다. 이번 계획은 기존 기능을 건드리지 않고, 역할별 파일로 분리해 향후 객실, 식음, 시설, 구매 모듈까지 확장 가능한 구조를 만드는 방향입니다.

현재 주 파일 1개 HTML
현재 코드 규모 약 5,900줄
리팩토링 방식 단계 분리

핵심 원칙

  • 원칙 1기능 변경 없이 파일 구조만 먼저 나눕니다.
  • 원칙 2각 단계 후 엑셀 업로드, 저장, 캘린더, AI, 자산관리를 확인합니다.
  • 원칙 3공통 로직은 호텔 ERP 플랫폼 공용 레이어로 이동합니다.
  • 원칙 4모든 파일은 UTF-8 기준으로 저장합니다.

단계별 리팩토링 순서

작게 나누어 진행해야 기능이 깨졌을 때 원인을 바로 찾을 수 있습니다.

  1. CSS 분리

    style 태그 내용을 `src/styles/app.css`로 이동하고 HTML에서 link로 불러옵니다. 기능 로직에는 손대지 않습니다.

    위험도 낮음

  2. 상수와 유틸 함수 분리

    색상, 식사유형, 라벨, 날짜/문자열/금액 유틸을 `constants.js`, `utils.js`로 이동합니다.

    위험도 중간

  3. Supabase 저장 계층 분리

    DB와 Storage 호출을 `storage.js` 또는 `services/supabaseClient.js`로 모읍니다. 화면 코드는 저장 함수만 호출하게 만듭니다.

    위험도 중간

  4. 로그인 모듈 분리

    내부 계정, currentUser, 관리자 권한, localStorage 세션 처리를 `auth.js`로 이동합니다.

    위험도 낮음

  5. 캘린더 모듈 분리

    월간 캘린더, 멀티데이 바, 행사 클릭, 상세 모달 연결을 `eventCalendar.js`로 이동합니다.

    위험도 높음

  6. 이벤트오더 추출 모듈 분리

    엑셀 파서와 추출 로직을 `eventOrder.js`와 `excelService.js`로 나눕니다. 가장 복잡하므로 테스트 파일을 함께 준비합니다.

    위험도 높음

  7. 자산관리와 AI 모듈 분리

    `assetManager.js`, `aiAssistant.js`로 분리하고, 각 모듈은 필요한 DOM과 service 함수만 주입받게 합니다.

    위험도 중간

  8. 플랫폼형 폴더로 정리

    연회 모듈을 `modules/banquet` 아래로 이동하고, 이후 객실/식음/시설/구매 모듈을 같은 방식으로 추가할 수 있게 합니다.

    위험도 중간

단계별 확인 테스트

각 분리 단계 후 아래 항목을 확인하면 회귀 오류를 빠르게 잡을 수 있습니다.

기본 동작 테스트

  • 로그인 전 ERP 화면이 숨겨지고, 로그인 후 표시되는지 확인
  • 관리자와 일반 사용자 권한에 따라 자산 버튼 표시가 달라지는지 확인
  • 새로고침 후 로그인 상태가 유지되는지 확인
  • 한글 라벨이 깨지지 않고 표시되는지 확인

행사 업무 테스트

  • 엑셀 업로드 후 행사명, 행사일시, 장소, Schedule, Items가 추출되는지 확인
  • 식사유형 체크박스와 행사유형 input이 저장되는지 확인
  • 저장 후 캘린더에 멀티데이 바가 이어져 보이는지 확인
  • 행사 상세 모달에서 수정, 삭제, 원본 엑셀 보기가 동작하는지 확인

Supabase 테스트

  • event_orders, calendar_dates, schedules, items, notes 저장 확인
  • Storage에 원본 엑셀과 자산 이미지가 업로드되는지 확인
  • 새로고침 후 Supabase DB에서 캘린더가 다시 로드되는지 확인
  • 저장 실패 시 실제 오류 메시지가 화면에 표시되는지 확인

AI와 다운로드 테스트

  • 업로드 직후 AI 자동 분석 카드가 표시되는지 확인
  • AI 채팅 질문이 Edge Function으로 전달되는지 확인
  • 자산관리 데이터가 AI 답변에 반영되는지 확인
  • 기간별 엑셀 다운로드 파일이 정상 생성되는지 확인

확장 설계 방향

연회 ERP를 호텔 ERP 플랫폼으로 확장할 때 필요한 경계입니다.

플랫폼 계층 담당 범위 연회 모듈 예시 다른 부서 확장 예시
core 앱 초기화, 탭/라우팅, 전역 상태 연회 캘린더 탭 연결 객실, 식음, 시설, 구매 메뉴 추가
services Supabase, Storage, Excel, AI API 호출 event_orders 저장 room_orders, purchase_requests 저장
shared 공통 상수, 날짜, 금액, 테이블 UI 행사 색상, 식사유형 부서 공통 코드, 상태값, 권한
modules 부서별 업무 화면과 로직 banquet/eventOrder, banquet/calendar rooms/housekeeping, fnb/reservation, engineering/workOrder

학습용 설계 기준

파일을 나누는 기준은 “코드 길이”가 아니라 “변경 이유가 같은 코드끼리 묶는 것”입니다.

왜 파일을 나누나

한 파일 안에 화면, 데이터, 업무 규칙이 모두 있으면 작은 수정도 전체 앱을 흔들 수 있습니다. 실무에서는 변경 가능성이 같은 코드끼리 묶어서 영향 범위를 작게 만듭니다.

변경 범위 축소 오류 추적 쉬움 협업 가능

왜 기능별로 나누나

로그인 수정과 엑셀 추출 수정은 서로 다른 업무 이유로 바뀝니다. 그래서 auth.js와 eventOrder.js처럼 기능 경계를 기준으로 분리합니다.

업무 기준 책임 분리 테스트 단순화

왜 공통 파일이 필요한가

날짜 포맷, 금액 포맷, 문자열 정리 같은 코드는 연회뿐 아니라 객실, 식음, 시설에서도 다시 쓰입니다. 이런 코드는 utils.js, constants.js로 빼서 플랫폼 자산으로 만듭니다.

중복 제거 호텔 공통 규칙 확장 대비

파일별 교육 메모

실제 분리 작업을 할 때 각 파일 상단에 남길 설명 주석의 방향입니다.

파일 왜 만드는가 이동할 코드 기준 실무 설계 이유
index.html 앱의 뼈대만 남기기 위해 만듭니다. 헤더, 메인 컨테이너, 모달, script/css import HTML은 구조를 담당하고, 업무 로직은 JS 모듈에서 관리하는 편이 안전합니다.
styles.css 디자인 변경이 기능 로직과 섞이지 않게 합니다. style 태그 전체, 반응형, 카드/테이블/캘린더 스타일 디자인 수정자는 JS를 건드리지 않고 UI만 조정할 수 있습니다.
auth.js 로그인과 권한 판단을 한곳에서 관리합니다. loginAccounts, currentUser, 로그인/로그아웃, 관리자 판별 권한 규칙은 보안과 연결되므로 화면 코드에 흩어지면 위험합니다.
eventCalendar.js 캘린더 표시와 행사 클릭 동작을 독립시킵니다. 월 이동, 날짜 계산, 멀티데이 바, 상세 모달 열기 캘린더는 화면 로직이 많아서 별도 모듈로 두면 버그 추적이 쉬워집니다.
eventOrder.js 이벤트오더 추출 규칙을 업무 도메인으로 분리합니다. 엑셀 업로드, 기본정보 추출, Schedule/Items/섹션 추출 호텔 양식이 바뀔 때 이 파일만 수정하면 되도록 경계를 만듭니다.
assetManager.js 자산관리 업무를 연회 행사 로직과 분리합니다. 자산 조회, 등록, 수정, 삭제, 이미지 모달 나중에 객실/시설 자산으로 확장할 때 같은 패턴을 재사용할 수 있습니다.
aiAssistant.js AI 채팅과 자동 분석 UI를 한곳에서 관리합니다. AI 패널, 자동 분석 카드, Edge Function 호출 AI 기능은 변경 속도가 빠르므로 다른 업무 화면과 분리하는 것이 좋습니다.
storage.js Supabase DB/Storage 호출을 표준화합니다. select, insert, update, delete, 파일 업로드, public URL 생성 DB 스키마가 바뀌어도 화면 파일을 대량 수정하지 않게 해줍니다.
constants.js 색상, 식사유형, 고정 라벨을 한곳에 둡니다. 색상칩, 식사유형 옵션, 권한값, 기본 메시지 같은 값을 여러 파일에 복사하면 나중에 서로 달라지는 문제가 생깁니다.
utils.js 업무와 무관한 작은 도구 함수를 모읍니다. 날짜 포맷, 문자열 정리, 금액 변환, DOM 헬퍼 연회뿐 아니라 다른 호텔 모듈에서도 반복 사용할 수 있습니다.

주석 작성 원칙

교육용 리팩토링에서는 “무엇을 하는지”보다 “왜 이 위치에 있는지”를 주석으로 남깁니다.

좋은 주석

  • 이 파일이 담당하는 업무 범위를 설명합니다.
  • 다른 파일이 이 파일을 어떻게 호출해야 하는지 설명합니다.
  • 호텔 ERP 확장 시 어떤 부분을 재사용할 수 있는지 설명합니다.
  • 복잡한 추출 규칙처럼 실무 맥락이 필요한 곳에만 자세히 씁니다.

피해야 할 주석

학습용 진행 방식

다음 단계에서 실제 파일 분리를 진행할 때는 매 단계마다 주석과 확인 기록을 같이 남깁니다.

단계 작업 남길 주석 학습 포인트
1 CSS 분리 테마 변수와 레이아웃 규칙의 목적 디자인 변경과 기능 변경을 분리하는 이유
2 상수 분리 색상/식사유형/권한값을 한곳에 두는 이유 중복된 값이 실무에서 버그가 되는 과정
3 저장 계층 분리 DB 호출을 화면에서 숨기는 이유 서비스 레이어와 화면 레이어의 경계
4 이벤트오더 파서 분리 호텔 양식별 fallback 추출 규칙 업무 규칙을 도메인 모듈로 모으는 이유
5 캘린더/모달 분리 화면 상태와 데이터 상태를 구분하는 이유 UI 컴포넌트 구조를 잡는 방법
6 AI/자산관리 분리 빠르게 변하는 기능을 독립시키는 이유 모듈별 변경 속도에 따라 파일을 나누는 방법

실제 리팩토링을 시작하면 각 파일 첫머리에 “역할”, “이 파일로 이동한 이유”, “다른 파일과의 경계”를 주석으로 남기겠습니다. 복잡한 함수 안에는 업무 규칙을 설명하는 주석만 추가하고, 단순한 코드 설명 주석은 넣지 않는 방식으로 진행합니다.

이 문서는 기존 ERP 코드를 수정하지 않고 작성한 리팩토링 계획서입니다. 실제 분리 작업은 CSS 분리부터 시작해 한 단계씩 검증하는 방식이 가장 안전합니다.
